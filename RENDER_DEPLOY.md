# Deploying Vijayavipanchi to Render.com

This deploys the portal as **two Render web services** from one repo, using the
included [`render.yaml`](./render.yaml) Blueprint:

| Service | What it is | Runs |
|---|---|---|
| `vijayavipanchi-api` | Express + TypeScript API | `npm run start:prod` (tsx) on Render's injected `PORT` |
| `vijayavipanchi-web` | Next.js 15 frontend | `next start`; proxies `/api` + `/media` to the API |

Because Render has **no managed MongoDB** and a **read‑only‑after‑deploy / ephemeral
filesystem**, two external (free) services are used:

- **MongoDB Atlas** (free M0 cluster) — the database.
- **Cloudinary** (free tier) — stores uploaded **and seeded** audio/images so they
  survive redeploys. (Without it, uploads would vanish on the next deploy/restart.)

> The frontend talks only to its own origin; the Next server proxies `/api` and
> `/media` to the API, so cookies + CSRF keep working over a single HTTPS origin.

---

## 0. Prerequisites

- A **GitHub** (or GitLab) account — Render deploys from a Git repo.
- A free **Render** account — <https://render.com>
- A free **MongoDB Atlas** account — <https://www.mongodb.com/atlas>
- A free **Cloudinary** account — <https://cloudinary.com>
- Node 20+ locally (only needed to run the one‑time seed).

---

## 1. Push the repo to GitHub

From the project root:

```bash
git add -A
git commit -m "Add Render deployment config (Cloudinary + Atlas)"
# create an empty repo on GitHub first, then:
git remote add origin https://github.com/<you>/vijayavipanchi.git   # skip if already added
git branch -M main
git push -u origin main
```

## 2. Create the MongoDB Atlas cluster

1. Atlas → **Build a Database → M0 (Free)** → pick a region → create.
2. **Database Access** → add a user (username + password). Save the password.
3. **Network Access** → **Add IP Address** → `0.0.0.0/0` (allow from anywhere — Render's
   egress IPs aren't fixed on the free tier).
4. **Connect → Drivers** → copy the **SRV** connection string, e.g.
   `mongodb+srv://USER:PASSWORD@cluster0.xxxx.mongodb.net/?retryWrites=true&w=majority`
5. Insert the database name **`swara`** before the `?`:
   `mongodb+srv://USER:PASSWORD@cluster0.xxxx.mongodb.net/swara?retryWrites=true&w=majority`
   This is your **`MONGODB_URI`**.

## 3. Get Cloudinary credentials

Cloudinary **Dashboard** shows **Cloud name**, **API Key**, **API Secret**. You'll use
these as `CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, `CLOUDINARY_API_SECRET`.

> If seeded cover images don't appear later, enable **Settings → Security → Allow
> delivery of SVG files** in Cloudinary (the generated covers are SVG).

## 4. Deploy the Blueprint on Render

1. Render Dashboard → **New → Blueprint**.
2. Connect your GitHub repo. Render detects `render.yaml` and lists **two services**.
3. Render prompts for every `sync: false` value. Fill the **API** service:
   - `MONGODB_URI` — from step 2
   - `ADMIN_EMAIL` — your email · `ADMIN_PASSWORD` — a strong password
   - `CLOUDINARY_CLOUD_NAME` / `CLOUDINARY_API_KEY` / `CLOUDINARY_API_SECRET` — step 3
   - `CORS_ORIGIN` — leave a placeholder for now (set in step 5)
   - SMTP_* — optional (only for admin email replies); leave blank to skip
   - `JWT_SECRET` and `CAPTCHA_SECRET` are auto‑generated by Render.
4. **Apply** — Render builds and starts both services. Note the two URLs it assigns,
   e.g. `https://vijayavipanchi-api.onrender.com` and `https://vijayavipanchi-web.onrender.com`.

## 5. Wire the two services together

After both URLs exist, set the cross‑references and redeploy:

- **vijayavipanchi-web** → Environment → set both:
  - `API_PROXY_TARGET = https://vijayavipanchi-api.onrender.com`
  - `API_INTERNAL_URL = https://vijayavipanchi-api.onrender.com`
- **vijayavipanchi-api** → Environment → set:
  - `CORS_ORIGIN = https://vijayavipanchi-web.onrender.com`

Save each (Render redeploys the affected service automatically).

## 6. Seed the database (one time)

The seed creates the categories, songs (with multi‑language lyrics), the first admin
user, and uploads the generated cover/audio media to Cloudinary. Run it **locally**,
pointed at Atlas + Cloudinary (it resets the `swara` database):

```bash
cd backend
MONGODB_URI="mongodb+srv://USER:PASSWORD@cluster0.xxxx.mongodb.net/swara?retryWrites=true&w=majority" \
CLOUDINARY_CLOUD_NAME="your_cloud" \
CLOUDINARY_API_KEY="your_key" \
CLOUDINARY_API_SECRET="your_secret" \
ADMIN_USERNAME="admin" \
ADMIN_EMAIL="you@example.com" \
ADMIN_PASSWORD="YourStrongPassword" \
npm run seed
```

(Use the **same** `ADMIN_*` values you set on Render so the login matches.)

## 7. Verify

1. Open the **frontend** URL → the homepage should show songs, covers, and a playable
   audio sample (served from Cloudinary).
2. Go to `/admin/login` → sign in with your `ADMIN_USERNAME` / `ADMIN_PASSWORD`.
3. Upload a song with a cover/audio in the admin → confirm it appears and plays
   (it's now stored on Cloudinary and survives redeploys).

---

## Notes & gotchas

- **Free‑tier cold starts:** free services sleep after ~15 min idle; the first request
  then takes ~30–60s. Upgrade either service to **Starter** to keep it always on.
- **First admin only via seed.** After deploy, create/edit more admins in
  **Admin → Users** (no need to re‑seed).
- **Changing the admin password** later: do it in **Admin → Users** (don't rely on
  `ADMIN_PASSWORD`, which only seeds the first account).
- **Custom domain:** add it on the **frontend** service, then update `CORS_ORIGIN`
  (API) and `API_PROXY_TARGET`/`API_INTERNAL_URL` (web) to the new domain.
- **Lyrics PDF (Indic scripts):** Telugu/Devanagari glyph shaping needs the Noto fonts
  in `backend/assets/fonts/` (fetched by `scripts/fetch-fonts.mjs`); without them the
  PDF falls back to Latin text. Commit the fonts or run that script before deploy if
  you need fully shaped Indic PDFs.
- **Local dev is unchanged:** leave the `CLOUDINARY_*` vars blank and media is stored
  on local disk under `DATA_DIR` exactly as before.
